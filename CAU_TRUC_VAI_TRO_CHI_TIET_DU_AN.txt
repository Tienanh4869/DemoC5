GIAI THICH CHI TIET TUNG FILE & VAI TRO TRONG DU AN

--------------------------------------------------------------------------------
NHOM 1: TANG TU DONG HOA CI/CD & SLSA PROVENANCE (.github/workflows/)
--------------------------------------------------------------------------------
1. File: .github/workflows/slsa-cosign-pipeline.yml
- NÓ LÀ GÌ: Là kịch bản định nghĩa luồng tự động hóa CI/CD (Continuous Integration / Continuous Delivery) chạy trên máy chủ ảo của GitHub Actions mỗi khi có lệnh git push.
- DÙNG ĐỂ LÀM GÌ: Tự động hóa 100% quy trình 6 bước: Đăng nhập Azure ACR -> Build Docker Image -> Xuất SBOM Syft -> Ký số Cosign -> Push lên Cloud Registry.
- KẾT NỐI DỊCH VỤ / CÔNG CỤ NÀO: Máy chủ GitHub Actions Runner, Azure Container Registry (demobyta.azurecr.io), công cụ Syft và Cosign.
- THUỘC TRỤ CỘT NÀO: Trụ cột 1 (SLSA Provenance Level 3), Trụ cột 2 (SBOM) và Trụ cột 3 (Sigstore Cosign).

--------------------------------------------------------------------------------
NHOM 2: TANG DOI TUONG DICH MAU & BAO MAT CORE CONTAINER (app/)
--------------------------------------------------------------------------------
2. File: app/Dockerfile
- NÓ LÀ GÌ: Kịch bản hướng dẫn Docker đóng gói mã nguồn ứng dụng thành Container Image chuẩn bảo mật.
- DÙNG ĐỂ LÀM GÌ: Áp dụng triết lý "Secure by Design" bằng cách: sử dụng ảnh gốc siêu nhẹ nginx:1.26-alpine (giảm tối đa bề mặt tấn công) và cấu hình chạy dưới quyền người dùng Non-Root (USER nginx) thay vì quyền root. Ngăn chặn triệt để lỗ hổng leo thang đặc quyền nếu bị xâm nhập.
- KẾT NỐI DỊCH VỤ / CÔNG CỤ NÀO: Docker Engine, Azure Container Registry (ACR), Azure Kubernetes Service (AKS).
- THUỘC TRỤ CỘT NÀO: Bảo mật nền tảng lõi Container & SLSA Build.

3. File: app/nginx.conf
- NÓ LÀ GÌ: File cấu hình thông số hoạt động của máy chủ Web Nginx bên trong Container.
- DÙNG ĐỂ LÀM GÌ: Cấu hình Nginx lắng nghe ở cổng bảo mật 8080 (vì tài khoản Non-Root không có quyền mở các cổng dưới 1024 như cổng 80), đồng thời tạo ra endpoint "/healthz" phục vụ cơ chế tự chẩn đoán liveness/readiness probe của Kubernetes.
- KẾT NỐI DỊCH VỤ / CÔNG CỤ NÀO: Nginx, Kubernetes Healthcheck Probes trên AKS.
- THUỘC TRỤ CỘT NÀO: Bảo mật nền tảng lõi & Vận hành Cloud-Native.

4. File: app/index.html / style.css / script.js
- NÓ LÀ GÌ: Giao diện Web Dashboard tượng trưng (Symbolic Proof UI) được hiển thị ra trình duyệt khi người dùng truy cập.
- DÙNG ĐỂ LÀM GÌ: Đóng vai trò là "Bằng chứng sống" (Proof of Service Delivery). Minh chứng rằng: Pod sau khi vượt qua quy trình kiểm duyệt Zero Trust khắt khe của Kyverno vẫn vận hành mượt mà, hiển thị rõ chứng nhận PASSED ZERO TRUST ADMISSION CONTROL và 3 Trụ cột SLSA - SBOM - Sigstore.
- KẾT NỐI DỊCH VỤ / CÔNG CỤ NÀO: Trình duyệt Web Chrome/Edge, cổng 8080 qua kubectl port-forward.
- THUỘC TRỤ CỘT NÀO: Trình diễn tổng hợp kết quả đồ án.

--------------------------------------------------------------------------------
NHOM 3: TANG MAT MA BAN QUYEN SIGSTORE COSIGN (keys/)
--------------------------------------------------------------------------------
5. File: keys/cosign.key
- NÓ LÀ GÌ: Private Key (Khóa bí mật / Khóa riêng) theo chuẩn mã hóa mật mã bất đối xứng ECDSA do công cụ Cosign sinh ra.
- DÙNG ĐỂ LÀM GÌ: Đóng vai trò như "Con dấu bản quyền tối cao" của tổ chức. Chỉ được giữ mật trong hệ thống CI/CD để ký (sign) lên Container Image và SBOM ngay sau khi build thành công. Bất kỳ sự thay đổi dù chỉ 1 bit sau khi ký đều làm mất hiệu lực chữ ký.
- KẾT NỐI DỊCH VỤ / CÔNG CỤ NÀO: Sigstore Cosign CLI, GitHub Secrets (COSIGN_PRIVATE_KEY).
- THUỘC TRỤ CỘT NÀO: Trụ cột 3 (Sigstore Cosign - Tính toàn vẹn Integrity).

6. File: keys/cosign.pub
- NÓ LÀ GÌ: Public Key (Khóa công khai) cặp đi kèm với cosign.key.
- DÙNG ĐỂ LÀM GÌ: Khóa này được nạp thẳng vào bộ nhớ của máy chủ Azure Kubernetes Service (AKS) và nạp cho "Người gác cổng" Kyverno cầm. Khi Pod xin khởi chạy, Kyverno dùng khóa công khai này để giải mã và xác minh xem chữ ký trên Azure Registry có đúng do chủ nhân của cosign.key ký hay không.
- KẾT NỐI DỊCH VỤ / CÔNG CỤ NÀO: Kyverno Admission Controller, Azure Kubernetes Service (AKS).
- THUỘC TRỤ CỘT NÀO: Trụ cột 3 (Sigstore Cosign) & Trụ cột 4 (Zero Trust Enforcement).

--------------------------------------------------------------------------------
NHOM 4: TANG THUC THI PHAO DAI ZERO TRUST KUBERNETES (policies/)
--------------------------------------------------------------------------------
7. File: policies/verify-cosign-policy.yaml
- NÓ LÀ GÌ: Manifest Kubernetes sử dụng Custom Resource Definition (CRD) loại ClusterPolicy của Kyverno Admission Controller.
- DÙNG ĐỂ LÀM GÌ: Đây là "Luật Zero Trust tối cao" trên cụm AKS. Cấu hình chặn ngay tại cổng vào (Admission Webhook): Bắt buộc mọi Pod (*) muốn chạy trên cụm đều PHẢI có chữ ký mật mã khớp với khóa công khai cosign.pub. Thiết lập "validationFailureAction: Enforce" để tiêu diệt ngay lập tức (DENY) nếu không có chữ ký hoặc chữ ký sai.
- KẾT NỐI DỊCH VỤ / CÔNG CỤ NÀO: Kyverno Admission Controller, máy chủ Azure Kubernetes Service (AKS).
- THUỘC TRỤ CỘT NÀO: Trụ cột 4 (Zero Trust Admission Control).

8. File: policies/deploy-valid.yaml
- NÓ LÀ GÌ: Manifest Kubernetes định nghĩa Deployment cho ứng dụng hợp lệ chính chủ (cloud-supply-chain-demo).
- DÙNG ĐỂ LÀM GÌ: Sử dụng Container Image đã được ký số hợp lệ kèm theo Chính xác mã băm SHA256 bất biến (@sha256:a456...). Khi apply file này, Kyverno kiểm tra chữ ký hợp lệ 100% nên CHO PHÉP (ALLOW) Pod khởi chạy thành công (Ready 1/1).
- KẾT NỐI DỊCH VỤ / CÔNG CỤ NÀO: Azure Container Registry (demobyta), Azure Kubernetes Service (AKS).
- THUỘC TRỤ CỘT NÀO: Kịch bản chứng minh cho phép ứng dụng sạch (ALLOW).

9. File: policies/deploy-invalid.yaml
- NÓ LÀ GÌ: Manifest Kubernetes định nghĩa Deployment cho ứng dụng kiểm thử tấn công (demo-invalid-unsigned).
- DÙNG ĐỂ LÀM GÌ: Cố tình ra lệnh cho AKS chạy một Image chuẩn từ mạng nhưng KHÔNG CÓ CHỮ KÝ của nhóm (unauthorized-hacker-app:latest). Khi apply file này, Kyverno lập tức CHẶN ĐỨNG (DENY), tiêu diệt Pod ngay từ trứng nước khiến Deployment ở trạng thái cảnh báo 0/1 Pod.
- KẾT NỐI DỊCH VỤ / CÔNG CỤ NÀO: Kyverno Policy Engine trên Azure AKS.
- THUỘC TRỤ CỘT NÀO: Kịch bản chứng minh tiêu diệt virus/hacker (DENY).

--------------------------------------------------------------------------------
NHOM 5: BO KICH BAN TU DONG HOA POWERSHELL (scripts/)
--------------------------------------------------------------------------------
10. File: scripts/01-setup-azure.ps1
- NÓ LÀ GÌ: Kịch bản PowerShell tự động hóa chuẩn bị hạ tầng đám mây Azure.
- DÙNG ĐỂ LÀM GÌ: Tự động đăng nhập Azure CLI, tạo Nhóm tài nguyên (rg-cloud-supply-chain-demo), tạo Registry (demobyta) và tạo Cụm Kubernetes AKS (aks-supply-chain-demo). Sau đó kết nối quyền truy cập (az aks update --attach-acr) để AKS có quyền tự động kéo Image từ ACR.
- KẾT NỐI DỊCH VỤ / CÔNG CỤ NÀO: Azure CLI (az), Azure Resource Group, ACR, AKS.
- THUỘC TRỤ CỘT NÀO: Hạ tầng nền tảng Cloud-Native.

11. File: scripts/02-build-sign.ps1
- NÓ LÀ GÌ: Kịch bản PowerShell tự động hóa luồng Build - Quét SBOM - Ký số Cosign trực tiếp từ laptop cá nhân.
- DÙNG ĐỂ LÀM GÌ: Đóng vai trò như luồng CI/CD cục bộ (giúp báo cáo nhanh tại lớp không cần chờ GitHub). Gọi lệnh "az acr build" đóng gói trên Cloud, gọi Syft quét xuất file SBOM "cloud-demo-sbom.json" (SPDX-JSON), và gọi Cosign dùng khóa riêng cosign.key ký số niêm phong Image trên Registry.
- KẾT NỐI DỊCH VỤ / CÔNG CỤ NÀO: Azure ACR Builder, Syft CLI, Cosign CLI.
- THUỘC TRỤ CỘT NÀO: Trụ cột 1, 2 và 3.

12. File: scripts/03-enforce-policy.ps1
- NÓ LÀ GÌ: Kịch bản PowerShell thực chiến kiểm thử pháo đài Zero Trust trên Kubernetes.
- DÙNG ĐỂ LÀM GÌ: Tự động cài đặt Kyverno vào AKS bằng Helm Chart, nạp chính sách verify-cosign-policy.yaml, sau đó lần lượt chạy tự động 2 Test Case thực chiến: Test Case 1 (apply deploy-invalid.yaml -> bị CHẶN DENY) và Test Case 2 (apply deploy-valid.yaml -> CHO PHÉP ALLOW).
- KẾT NỐI DỊCH VỤ / CÔNG CỤ NÀO: Helm CLI, Kubectl CLI, Kyverno Engine, Azure AKS.
- THUỘC TRỤ CỘT NÀO: Trụ cột 4 (Zero Trust Enforcement).
